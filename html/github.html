Chapter 1 — Setup & Basics

Goal: install Git, create a GitHub account, initialise repositories, make commits, push/pull. This chapter assumes no prior knowledge.

1.1 Install Git

Install on Linux (Debian/Ubuntu):

sudo apt update
sudo apt install git -y
git --version

Install on macOS (Homebrew):

brew install git
git --version

Install on Windows: download the Git for Windows installer (https://git-scm.com) and run it.

1.2 Configure Git

git config --global user.name "Your Name"
git config --global user.email "you@example.com"
git config --global core.editor "code --wait"  # optional (VS Code)

1.3 Create a GitHub account & repository

Steps (brief): sign up at github.com → New repository → name it devops-demo → choose Public/Private → Initialize with README (optional).

1.4 Local repo → commit → push

mkdir devops-demo
cd devops-demo
git init
# create a file
echo "# DevOps Demo" > README.md
git add README.md
git commit -m "chore: add README"
# link remote (from GitHub repo page)
git remote add origin git@github.com:YOUR_USERNAME/devops-demo.git
# push the main branch
git push -u origin main

1.5 Example: Fixing a small typo

# make change
sed -i 's/Dovops/DevOps/' README.md
git add README.md
git commit -m "fix: typo in README"
git push

Chapter 2 — Branching & Merging

Goal: learn branching strategies used in DevOps: feature branches, trunk-based, GitFlow basics, resolving merge conflicts.

2.1 Create and switch branches

git checkout -b feature/nginx-setup
# work on files
git add .
git commit -m "feat: add nginx config"
# push branch
git push -u origin feature/nginx-setup

2.2 Merge into main (fast-forward vs. merge commit)

# on local main
git checkout main
git pull origin main
git merge --no-ff feature/nginx-setup
git push

Use --no-ff if you want an explicit merge commit to keep history of feature branches.

2.3 Rebase (when to use)

Rebase keeps a linear history. Use carefully in private branches.

# rebase feature branch on updated main
git checkout feature/nginx-setup
git fetch origin
git rebase origin/main
# resolve conflicts if any, then
git rebase --continue
git push --force-with-lease

2.4 Example conflict workflow

  1. Two devs modify same line in file.
  2. When you rebase or merge, Git will mark conflict sections with <<<<<<<, =======, >>>>>>>.
# open file, edit the conflict, remove markers
git add conflicted-file
git rebase --continue

Chapter 3 — Collaboration & Pull Requests

Goal: create PRs, request reviews, use protected branches, use reviewers & code owners, and squash commits when merging.

3.1 Forks vs branches

Open source: fork the repo & make branches in your fork, then open a PR to upstream. Internal orgs: create a branch in the same repo and open a PR directly.

3.2 Create a Pull Request (PR)

  1. Push feature branch to GitHub.
  2. On GitHub, click "Compare & pull request".
  3. Fill title, description, link issue, assign reviewers.

3.3 PR best practices (DevOps-focused)

  • Small, focused PRs that change one concern.
  • Include setup/test steps in PR description so CI replicates environment.
  • Link Jira/issue ticket IDs and deployment notes.

3.4 Review & merge strategies

Common strategies: squash-and-merge (clean history), merge commit (keeps branches), rebase-and-merge (linear history).

Security-sensitive changes?✔️/❌

Chapter 4 — GitHub Actions (CI/CD)

Goal: build a simple CI pipeline with GitHub Actions, run tests, build image, and deploy (example: deploy to DockerHub or push artifact to cloud).

4.1 Anatomy of a workflow

# .github/workflows/ci.yml
name: CI
on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Set up Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
      - name: Install deps
        run: npm ci
      - name: Run tests
        run: npm test

4.2 Example: Build Docker image & push to Docker Hub

jobs:
  build-and-push:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Login to DockerHub
        uses: docker/login-action@v2
        with:
          username: ${{ secrets.DOCKERHUB_USER }}
          password: ${{ secrets.DOCKERHUB_TOKEN }}
      - name: Build and push
        uses: docker/build-push-action@v4
        with:
          push: true
          tags: youruser/yourimage:latest

4.3 Secrets & environment protection

Store credentials in repository or organization Secrets. Use required reviewers and environment protection rules before deployment jobs run.

Chapter 5 — Advanced Git (rewriting history, bisect, submodules)

5.1 Interactive rebase (cleaning commits)

git checkout feature/x
git rebase -i origin/main
# mark commits as pick / squash / fixup to combine them
# finish, then push with --force-with-lease

5.2 git bisect (finding a bad commit)

git bisect start
git bisect bad           # current bad rev
git bisect good v1.2.3   # known good rev
# git will checkout middle commit; test; mark good/bad until location found
git bisect reset

5.3 Submodules & monorepos

# add submodule
git submodule add https://github.com/yourorg/somelib.git libs/somelib
git submodule update --init --recursive

Chapter 6 — Security & Best Practices (DevOps)

  • Use branch protection rules for main and release branches.
  • Use CODEOWNERS to require specific reviewers for sensitive paths.
  • Audit repository secrets and rotate tokens regularly.
  • Enable Dependabot for dependency updates and security alerts.
