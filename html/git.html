1. What is Git?

Git is a Distributed Version Control System (DVCS) that helps developers track changes in code, collaborate with others, and revert to previous versions easily.

Key Features:

  • Tracks changes in source code with detailed history
  • Enables teamwork via branches and merging
  • Distributed - every user has a full copy
  • Offline capability - works without internet
  • Commit history - every version can be rolled back

2. What is GitHub?

GitHub is a cloud-based hosting platform for Git repositories. It helps you store, share, and collaborate on Git projects online with teams around the world.

Key Features:

  • Remote repository hosting - store projects in the cloud
  • Collaboration via Pull Requests and code reviews
  • Access control - manage who can view/edit code
  • CI/CD integration - automated testing and deployment
  • Integration with tools - VS Code, Docker, Jenkins, etc.
üí° Think of it like:
‚Ä¢ Git = Local tool for version control on your machine
‚Ä¢ GitHub = Online home for your Git projects (remote storage)

7. Interview Questions

Q: What is Git and why is it important?

A: Git is a distributed version control system that tracks changes in source code. It enables team collaboration, maintains complete history, allows rollback to previous versions, prevents data loss with multiple copies, and supports parallel development through branching.
Q: What is the difference between Git and GitHub?

A: Git = A version control tool that runs locally on your computer. GitHub = A cloud platform that hosts Git repositories for collaboration.
Q: What is a branch and why do we use branches?

A: A branch is a separate line of development. We use branches to work on new features without affecting main code, allow multiple developers to work independently, test code before merging into production, and organize work by feature or bugfix.
Q: What is the difference between merge, rebase, and reset?

A: Merge = Combines branches, creates merge commit, preserves history. Rebase = Replays commits linearly, rewrites history, cleaner but risky. Reset = Moves HEAD pointer, undoes commits, only for local use.
Q: What happens during a merge conflict?

A: A merge conflict occurs when two branches modified the same lines in a file differently. You must manually edit the file to resolve conflict, remove conflict markers, stage the resolved file, and commit the merge.
Q: What is a Pull Request (PR)?

A: A Pull Request is a request to merge changes from one branch into another. It allows team members to review code before merging, discuss changes, run automated tests, and maintain code quality.
Q: How do you undo the last commit?

A: Option 1 (Rewrite History - Local Only): git reset --soft HEAD~1 (keeps changes staged) or git reset --hard HEAD~1 (deletes changes). Option 2 (Safe - Creates New Commit): git revert HEAD (creates new commit that undoes the change).
Q: What is the difference between fetch and pull?

A: git fetch = Downloads changes from remote (doesn't merge). git pull = Downloads AND merges changes from remote (fetch + merge).
Q: What is rebase and how is it different from merge?

A: Rebase rewrites commit history to create linear sequence of commits. Merge = Non-linear history, creates merge commit, safe for shared branches. Rebase = Linear history, replays commits, cleaner but risky for public commits.
Q: What is the golden rule of rebasing?

A: "Never rebase public history". If commits are already pushed to shared repository and others have pulled them, DO NOT rebase. Use merge or revert instead.
